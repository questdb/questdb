79 /**
80  * A hash table supporting full concurrency of retrievals and
81  * high expected concurrency for updates. This class obeys the
82  * same functional specification as {@link java.util.Hashtable}, and
83  * includes versions of methods corresponding to each method of
84  * {@code Hashtable}. However, even though all operations are
85  * thread-safe, retrieval operations do <em>not</em> entail locking,
86  * and there is <em>not</em> any support for locking the entire table
87  * in a way that prevents all access.  This class is fully
88  * interoperable with {@code Hashtable} in programs that rely on its
89  * thread safety but not on its synchronization details.
90  * <p>Retrieval operations (including {@code get}) generally do not
91  * block, so may overlap with update operations (including {@code put}
92  * and {@code remove}). Retrievals reflect the results of the most
93  * recently <em>completed</em> update operations holding upon their
94  * onset. (More formally, an update operation for a given key bears a
95  * <em>happens-before</em> relation with any (non-null) retrieval for
96  * that key reporting the updated value.)  For aggregate operations
97  * such as {@code putAll} and {@code clear}, concurrent retrievals may
98  * reflect insertion or removal of only some entries.  Similarly,
99  * Iterators, Spliterators and Enumerations return elements reflecting the
100  * state of the hash table at some point at or since the creation of the
101  * iterator/enumeration.  They do <em>not</em> throw {@link
102  * java.util.ConcurrentModificationException ConcurrentModificationException}.
103  * However, iterators are designed to be used by only one thread at a time.
104  * Bear in mind that the results of aggregate status methods including
105  * {@code size}, {@code isEmpty}, and {@code containsValue} are typically
106  * useful only when a map is not undergoing concurrent updates in other threads.
107  * Otherwise the results of these methods reflect transient states
108  * that may be adequate for monitoring or estimation purposes, but not
109  * for program control.
110  * <p>The table is dynamically expanded when there are too many
111  * collisions (i.e., keys that have distinct hash codes but fall into
112  * the same slot modulo the table size), with the expected average
113  * effect of maintaining roughly two bins per mapping (corresponding
114  * to a 0.75 load factor threshold for resizing). There may be much
115  * variance around this average as mappings are added and removed, but
116  * overall, this maintains a commonly accepted time/space tradeoff for
117  * hash tables.  However, resizing this or any other kind of hash
118  * table may be a relatively slow operation. When possible, it is a
119  * good idea to provide a size estimate as an optional {@code
120  * initialCapacity} constructor argument. An additional optional
121  * {@code loadFactor} constructor argument provides a further means of
122  * customizing initial table capacity by specifying the table density
123  * to be used in calculating the amount of space to allocate for the
124  * given number of elements.  Also, for compatibility with previous
125  * versions of this class, constructors may optionally specify an
126  * expected {@code concurrencyLevel} as an additional hint for
127  * internal sizing.  Note that using many keys with exactly the same
128  * {@code hashCode()} is a sure way to slow down performance of any
129  * hash table. To ameliorate impact, when keys are {@link Comparable},
130  * this class may use comparison order among keys to help break ties.
131  * <p>A {@link Set} projection of a ConcurrentHashMap may be created
132  * (using {@link #newKeySet()} or {@link #newKeySet(int)}), or viewed
133  * (using {@link #keySet(Object)} when only keys are of interest, and the
134  * mapped values are (perhaps transiently) not used or all take the
135  * same mapping value.
136  * <p> This class and its views and iterators implement all of the
137  * <em>optional</em> methods of the {@link Map} and {@link Iterator}
138  * interfaces.
139  * <p>Like {@link Hashtable} but unlike {@link HashMap}, this class
140  * does <em>not</em> allow {@code null} to be used as a key or value.
141  * <p>ConcurrentHashMaps support a set of sequential and parallel bulk
142  * operations that are designed
143  * to be safely, and often sensibly, applied even with maps that are
144  * being concurrently updated by other threads; for example, when
145  * computing a snapshot summary of the values in a shared registry.
146  * There are three kinds of operation, each with four forms, accepting
147  * functions with Keys, Values, Entries, and (Key, Value) arguments
148  * and/or return values. Because the elements of a ConcurrentHashMap
149  * are not ordered in any particular way, and may be processed in
150  * different orders in different parallel executions, the correctness
151  * of supplied functions should not depend on any ordering, or on any
152  * other objects or values that may transiently change while
153  * computation is in progress; and except for forEach actions, should
154  * ideally be side-effect-free. Bulk operations on {@link java.util.Map.Entry}
155  * objects do not support method {@code setValue}.
156  * <ul>
157  * <li> forEach: Perform a given action on each element.
158  * A variant form applies a given transformation on each element
159  * before performing the action.</li>
160  * <li> search: Return the first available non-null result of
161  * applying a given function on each element; skipping further
162  * search when a result is found.</li>
163  * <li> reduce: Accumulate each element.  The supplied reduction
164  * function cannot rely on ordering (more formally, it should be
165  * both associative and commutative).  There are five variants:
166  * <ul>
167  * <li> Plain reductions. (There is not a form of this method for
168  * (key, value) function arguments since there is no corresponding
169  * return type.)</li>
170  * <li> Mapped reductions that accumulate the results of a given
171  * function applied to each element.</li>
172  * <li> Reductions to scalar doubles, longs, and ints, using a
173  * given basis value.</li>
174  * </ul>
175  * </li>
176  * </ul>
177  * <p>The concurrency properties of bulk operations follow
178  * from those of ConcurrentHashMap: Any non-null result returned
179  * from {@code get(key)} and related access methods bears a
180  * happens-before relation with the associated insertion or
181  * update.  The result of any bulk operation reflects the
182  * composition of these per-element relations (but is not
183  * necessarily atomic with respect to the map as a whole unless it
184  * is somehow known to be quiescent).  Conversely, because keys
185  * and values in the map are never null, null serves as a reliable
186  * atomic indicator of the current lack of any result.  To
187  * maintain this property, null serves as an implicit basis for
188  * all non-scalar reduction operations. For the double, long, and
189  * int versions, the basis should be one that, when combined with
190  * any other value, returns that other value (more formally, it
191  * should be the identity element for the reduction). Most common
192  * reductions have these properties; for example, computing a sum
193  * with basis 0 or a minimum with basis MAX_VALUE.
194  * <p>Search and transformation functions provided as arguments
195  * should similarly return null to indicate the lack of any result
196  * (in which case it is not used). In the case of mapped
197  * reductions, this also enables transformations to serve as
198  * filters, returning null (or, in the case of primitive
199  * specializations, the identity basis) if the element should not
200  * be combined. You can create compound transformations and
201  * filterings by composing them yourself under this "null means
202  * there is nothing there now" rule before using them in search or
203  * reduce operations.
204  * <p>Methods accepting and/or returning Entry arguments maintain
205  * key-value associations. They may be useful for example when
206  * finding the key for the greatest value. Note that "plain" Entry
207  * arguments can be supplied using {@code new
208  * AbstractMap.SimpleEntry(k,v)}.
209  * <p>Bulk operations may complete abruptly, throwing an
210  * exception encountered in the application of a supplied
211  * function. Bear in mind when handling such exceptions that other
212  * concurrently executing functions could also have thrown
213  * exceptions, or would have done so if the first exception had
214  * not occurred.
215  * <p>Speedups for parallel compared to sequential forms are common
216  * but not guaranteed.  Parallel operations involving brief functions
217  * on small maps may execute more slowly than sequential forms if the
218  * underlying work to parallelize the computation is more expensive
219  * than the computation itself.  Similarly, parallelization may not
220  * lead to much actual parallelism if all processors are busy
221  * performing unrelated tasks.
222  * <p>All arguments to all task methods must be non-null.
223  * <p>This class is a member of the
224  * <a href="{@docRoot}/../technotes/guides/collections/index.html">
225  * Java Collections Framework</a>.
226  *
227  * @param <V> the type of mapped values
228  * @author Doug Lea
229  * @since 1.5
230  */

236     /*
237      * Overview:
238      *
239      * The primary design goal of this hash table is to maintain
240      * concurrent readability (typically method get(), but also
241      * iterators and related methods) while minimizing update
242      * contention. Secondary goals are to keep space consumption about
243      * the same or better than java.util.HashMap, and to support high
244      * initial insertion rates on an empty table by many threads.
245      *
246      * This map usually acts as a binned (bucketed) hash table.  Each
247      * key-value mapping is held in a Node.  Most nodes are instances
248      * of the basic Node class with hash, key, value, and next
249      * fields. However, various subclasses exist: TreeNodes are
250      * arranged in balanced trees, not lists.  TreeBins hold the roots
251      * of sets of TreeNodes. ForwardingNodes are placed at the heads
252      * of bins during resizing. ReservationNodes are used as
253      * placeholders while establishing values in computeIfAbsent and
254      * related methods.  The types TreeBin, ForwardingNode, and
255      * ReservationNode do not hold normal user keys, values, or
256      * hashes, and are readily distinguishable during search etc
257      * because they have negative hash fields and null key and value
258      * fields. (These special nodes are either uncommon or transient,
259      * so the impact of carrying around some unused fields is
260      * insignificant.)
261      *
262      * The table is lazily initialized to a power-of-two size upon the
263      * first insertion.  Each bin in the table normally contains a
264      * list of Nodes (most often, the list has only zero or one Node).
265      * Table accesses require volatile/atomic reads, writes, and
266      * CASes.  Because there is no other way to arrange this without
267      * adding further indirections, we use intrinsics
268      * (sun.misc.Unsafe) operations.
269      *
270      * We use the top (sign) bit of Node hash fields for control
271      * purposes -- it is available anyway because of addressing
272      * constraints.  Nodes with negative hash fields are specially
273      * handled or ignored in map methods.
274      *
275      * Insertion (via put or its variants) of the first node in an
276      * empty bin is performed by just CASing it to the bin.  This is
277      * by far the most common case for put operations under most
278      * key/hash distributions.  Other update operations (insert,
279      * delete, and replace) require locks.  We do not want to waste
280      * the space required to associate a distinct lock object with
281      * each bin, so instead use the first node of a bin list itself as
282      * a lock. Locking support for these locks relies on builtin
283      * "synchronized" monitors.
284      *
285      * Using the first node of a list as a lock does not by itself
286      * suffice though: When a node is locked, any update must first
287      * validate that it is still the first node after locking it, and
288      * retry if not. Because new nodes are always appended to lists,
289      * once a node is first in a bin, it remains first until deleted
290      * or the bin becomes invalidated (upon resizing).
291      *
292      * The main disadvantage of per-bin locks is that other update
293      * operations on other nodes in a bin list protected by the same
294      * lock can stall, for example when user equals() or mapping
295      * functions take a long time.  However, statistically, under
296      * random hash codes, this is not a common problem.  Ideally, the
297      * frequency of nodes in bins follows a Poisson distribution
298      * (http://en.wikipedia.org/wiki/Poisson_distribution) with a
299      * parameter of about 0.5 on average, given the resizing threshold
300      * of 0.75, although with a large variance because of resizing
301      * granularity. Ignoring variance, the expected occurrences of
302      * list size k are (exp(-0.5) * pow(0.5, k) / factorial(k)). The
303      * first values are:
304      *
305      * 0:    0.60653066
306      * 1:    0.30326533
307      * 2:    0.07581633
308      * 3:    0.01263606
309      * 4:    0.00157952
310      * 5:    0.00015795
311      * 6:    0.00001316
312      * 7:    0.00000094
313      * 8:    0.00000006
314      * more: less than 1 in ten million
315      *
316      * Lock contention probability for two threads accessing distinct
317      * elements is roughly 1 / (8 * #elements) under random hashes.
318      *
319      * Actual hash code distributions encountered in practice
320      * sometimes deviate significantly from uniform randomness.  This
321      * includes the case when N > (1<<30), so some keys MUST collide.
322      * Similarly for dumb or hostile usages in which multiple keys are
323      * designed to have identical hash codes or ones that differs only
324      * in masked-out high bits. So we use a secondary strategy that
325      * applies when the number of nodes in a bin exceeds a
326      * threshold. These TreeBins use a balanced tree to hold nodes (a
327      * specialized form of red-black trees), bounding search time to
328      * O(log N).  Each search step in a TreeBin is at least twice as
329      * slow as in a regular list, but given that N cannot exceed
330      * (1<<64) (before running out of addresses) this bounds search
331      * steps, lock hold times, etc, to reasonable constants (roughly
332      * 100 nodes inspected per operation worst case) so long as keys
333      * are Comparable (which is very common -- String, Long, etc).
334      * TreeBin nodes (TreeNodes) also maintain the same "next"
335      * traversal pointers as regular nodes, so can be traversed in
336      * iterators in the same way.
337      *
338      * The table is resized when occupancy exceeds a percentage
339      * threshold (nominally, 0.75, but see below).  Any thread
340      * noticing an overfull bin may assist in resizing after the
341      * initiating thread allocates and sets up the replacement array.
342      * However, rather than stalling, these other threads may proceed
343      * with insertions etc.  The use of TreeBins shields us from the
344      * worst case effects of overfilling while resizes are in
345      * progress.  Resizing proceeds by transferring bins, one by one,
346      * from the table to the next table. However, threads claim small
347      * blocks of indices to transfer (via field transferIndex) before
348      * doing so, reducing contention.  A generation stamp in field
349      * sizeCtl ensures that resizings do not overlap. Because we are
350      * using power-of-two expansion, the elements from each bin must
351      * either stay at same index, or move with a power of two
352      * offset. We eliminate unnecessary node creation by catching
353      * cases where old nodes can be reused because their next fields
354      * won't change.  On average, only about one-sixth of them need
355      * cloning when a table doubles. The nodes they replace will be
356      * garbage collectable as soon as they are no longer referenced by
357      * any reader thread that may be in the midst of concurrently
358      * traversing table.  Upon transfer, the old table bin contains
359      * only a special forwarding node (with hash field "MOVED") that
360      * contains the next table as its key. On encountering a
361      * forwarding node, access and update operations restart, using
362      * the new table.
363      *
364      * Each bin transfer requires its bin lock, which can stall
365      * waiting for locks while resizing. However, because other
366      * threads can join in and help resize rather than contend for
367      * locks, average aggregate waits become shorter as resizing
368      * progresses.  The transfer operation must also ensure that all
369      * accessible bins in both the old and new table are usable by any
370      * traversal.  This is arranged in part by proceeding from the
371      * last bin (table.length - 1) up towards the first.  Upon seeing
372      * a forwarding node, traversals (see class Traverser) arrange to
373      * move to the new table without revisiting nodes.  To ensure that
374      * no intervening nodes are skipped even when moved out of order,
375      * a stack (see class TableStack) is created on first encounter of
376      * a forwarding node during a traversal, to maintain its place if
377      * later processing the current table. The need for these
378      * save/restore mechanics is relatively rare, but when one
379      * forwarding node is encountered, typically many more will be.
380      * So Traversers use a simple caching scheme to avoid creating so
381      * many new TableStack nodes. (Thanks to Peter Levart for
382      * suggesting use of a stack here.)
383      *
384      * The traversal scheme also applies to partial traversals of
385      * ranges of bins (via an alternate Traverser constructor)
386      * to support partitioned aggregate operations.  Also, read-only
387      * operations give up if ever forwarded to a null table, which
388      * provides support for shutdown-style clearing, which is also not
389      * currently implemented.
390      *
391      * Lazy table initialization minimizes footprint until first use,
392      * and also avoids resizings when the first operation is from a
393      * putAll, constructor with map argument, or deserialization.
394      * These cases attempt to override the initial capacity settings,
395      * but harmlessly fail to take effect in cases of races.
396      *
397      * The element count is maintained using a specialization of
398      * LongAdder. We need to incorporate a specialization rather than
399      * just use a LongAdder in order to access implicit
400      * contention-sensing that leads to creation of multiple
401      * CounterCells.  The counter mechanics avoid contention on
402      * updates but can encounter cache thrashing if read too
403      * frequently during concurrent access. To avoid reading so often,
404      * resizing under contention is attempted only upon adding to a
405      * bin already holding two or more nodes. Under uniform hash
406      * distributions, the probability of this occurring at threshold
407      * is around 13%, meaning that only about 1 in 8 puts check
408      * threshold (and after resizing, many fewer do so).
409      *
410      * TreeBins use a special form of comparison for search and
411      * related operations (which is the main reason we cannot use
412      * existing collections such as TreeMaps). TreeBins contain
413      * Comparable elements, but may contain others, as well as
414      * elements that are Comparable but not necessarily Comparable for
415      * the same T, so we cannot invoke compareTo among them. To handle
416      * this, the tree is ordered primarily by hash value, then by
417      * Comparable.compareTo order if applicable.  On lookup at a node,
418      * if elements are not comparable or compare as 0 then both left
419      * and right children may need to be searched in the case of tied
420      * hash values. (This corresponds to the full list search that
421      * would be necessary if all elements were non-Comparable and had
422      * tied hashes.) On insertion, to keep a total ordering (or as
423      * close as is required here) across rebalancings, we compare
424      * classes and identityHashCodes as tie-breakers. The red-black
425      * balancing code is updated from pre-jdk-collections
426      * (http://gee.cs.oswego.edu/dl/classes/collections/RBCell.java)
427      * based in turn on Cormen, Leiserson, and Rivest "Introduction to
428      * Algorithms" (CLR).
429      *
430      * TreeBins also require an additional locking mechanism.  While
431      * list traversal is always possible by readers even during
432      * updates, tree traversal is not, mainly because of tree-rotations
433      * that may change the root node and/or its linkages.  TreeBins
434      * include a simple read-write lock mechanism parasitic on the
435      * main bin-synchronization strategy: Structural adjustments
436      * associated with an insertion or removal are already bin-locked
437      * (and so cannot conflict with other writers) but must wait for
438      * ongoing readers to finish. Since there can be only one such
439      * waiter, we use a simple scheme using a single "waiter" field to
440      * block writers.  However, readers need never block.  If the root
441      * lock is held, they proceed along the slow traversal path (via
442      * next-pointers) until the lock becomes available or the list is
443      * exhausted, whichever comes first. These cases are not fast, but
444      * maximize aggregate expected throughput.
445      *
446      * Maintaining API and serialization compatibility with previous
447      * versions of this class introduces several oddities. Mainly: We
448      * leave untouched but unused constructor arguments referring to
449      * concurrencyLevel. We accept a loadFactor constructor argument,
450      * but apply it only to initial table capacity (which is the only
451      * time that we can guarantee to honor it.) We also declare an
452      * unused "Segment" class that is instantiated in minimal form
453      * only when serializing.
454      *
455      * Also, solely for compatibility with previous versions of this
456      * class, it extends AbstractMap, even though all of its methods
457      * are overridden, so it is just useless baggage.
458      *
459      * This file is organized to make things a little easier to follow
460      * while reading than they might otherwise: First the main static
461      * declarations and utilities, then fields, then main public
462      * methods (with a few factorings of multiple public methods into
463      * internal ones), then sizing methods, trees, traversers, and
464      * bulk operations.
465      */

2135      * bit to 0. Because the table uses power-of-two masking, sets of
2136      * hashes that vary only in bits above the current mask will

2147      * never be used in index calculations because of table bounds.
2148      */
