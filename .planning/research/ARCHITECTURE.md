# Architecture Patterns

**Domain:** Exact percentile/quantile function hardening in QuestDB
**Researched:** 2026-04-13
**Confidence:** HIGH (all patterns verified via source code inspection on master and nw_percentile branch)

## Recommended Architecture

The percentile functions operate within QuestDB's existing group-by and window function frameworks. No new architectural components are needed -- the work fixes algorithms, memory management, and lifecycle correctness within the established structure. This document maps the exact patterns that existing (proven) functions follow, so the percentile hardening can follow the same patterns precisely.

### Component Boundaries

| Component | Responsibility | Key Files |
|-----------|---------------|-----------|
| GroupByFunction interface | Defines aggregate lifecycle (computeFirst/computeNext/merge/setNull/setEmpty) | `core/.../griffin/engine/functions/GroupByFunction.java` |
| WindowFunction interface | Defines window lifecycle (pass1/pass2/preparePass2/reset/reopen/toTop) | `core/.../griffin/engine/window/WindowFunction.java` |

## Pattern 1: Group-By Function Lifecycle

### Reference Implementation: `SumDoubleGroupByFunction`
**File:** `core/src/main/java/io/questdb/griffin/engine/functions/groupby/SumDoubleGroupByFunction.java`

The group-by function lifecycle has these phases, in order:

```
1. REGISTRATION
   initValueTypes(columnTypes) -- called once at compile time
     Registers how many MapValue slots the function needs.
     Example: Sum needs 1 DOUBLE slot.
     Percentile needs 1 LONG slot (pointer to GroupByDoubleList data).

2. INITIALIZATION
   setAllocator(GroupByAllocator) -- for functions using off-heap collections
     Called before any computation. The GroupByDoubleList/GroupByLongList
     use this allocator for malloc/free.

   init(symbolTableSource, executionContext) -- resolve constants
     Percentile functions resolve the percentile constant value here.

3. EMPTY STATE SETUP
   setEmpty(MapValue) -- called by generated bytecode via GroupByFunctionsUpdater
     Called when a new key appears in batch mode (computeBatch path).
     Default implementation calls setNull(). Functions that use 0 as
     "empty" (like Count) must override to distinguish "empty" from "null".
     
     CRITICAL: For percentile functions storing off-heap list pointers,
     setEmpty must set the pointer to 0 (meaning "no data yet"),
     NOT to LONG_NULL (which means "null result").

4. COMPUTATION
   computeFirst(MapValue, Record, rowId) -- first row for this group key
     Initializes aggregate state. For percentile: creates GroupByDoubleList,
     adds first value, stores pointer in MapValue.

   computeNext(MapValue, Record, rowId) -- subsequent rows for same key
     Accumulates. For percentile: flyweight-wraps pointer from MapValue,
     appends value, stores (potentially moved) pointer back.

5. PARALLEL MERGE
   merge(MapValue dest, MapValue src) -- merge partial results from workers
     Only called if supportsParallelism() returns true.
     For percentile: concatenates two GroupByDoubleLists.
     Both dest.isNew() and src.isNew() are guaranteed false.

6. RESULT EXTRACTION
   getDouble(Record) / getArray(Record) -- read from MapValue to produce result
     For percentile: wraps pointer, runs quickSelect, returns k-th element.

7. CLEANUP
   clear() -- reset between query executions (cursor reuse)
   close() -- release all resources (Misc.free on child functions)
```

### setEmpty vs setNull Distinction

**`setNull(MapValue)`** -- Sets the aggregate state to represent a NULL result. Used for SAMPLE BY FILL NULL and interpolation.
- Sum: `mapValue.putDouble(valueIndex, Double.NaN)`
- Percentile: `mapValue.putLong(valueIndex, Numbers.LONG_NULL)` -- means "result is NULL"

**`setEmpty(MapValue)`** -- Sets the aggregate state to "no data accumulated yet". Used by batch computation (computeBatch) where setEmpty is called first, then computeBatch accumulates.
- Default: calls setNull() -- correct for most functions
- Count: overrides to set 0 (not null)
- Percentile (BUG): should set pointer to 0 (no list allocated), NOT LONG_NULL

The distinction matters because computeBatch checks `mapValue.getLong(valueIndex)` to decide whether to create a new list or append to an existing one. LONG_NULL would be misinterpreted as a valid pointer.

### GroupByDoubleList Memory Pattern

The `GroupByDoubleList` (introduced on nw_percentile branch) is a flyweight wrapper over memory managed by `GroupByAllocator`:

```
Buffer layout:
+-------------------+------------------+----------------------------------+
| capacity (4 bytes)| size (4 bytes)   | double values (8 bytes each)     |
+-------------------+------------------+----------------------------------+

Usage pattern in computeFirst/computeNext:
  listA.of(0)              // of(0) = start new list (allocates via GroupByAllocator)
  listA.of(existingPtr)    // of(ptr) = wrap existing allocation
  listA.add(value)         // append; may realloc if capacity exceeded
  mapValue.putLong(valueIndex, listA.ptr())  // store (possibly moved) pointer back
```

The allocator is set once via `setAllocator()` and manages all memory. The function does NOT own the memory -- the allocator does. On `clear()`, call `listA.resetPtr()` and `listB.resetPtr()` to detach flyweight pointers (allocator handles actual freeing).

## Pattern 2: Window Function Two-Pass Architecture

### Orchestrator: `CachedWindowRecordCursorFactory.CachedWindowRecordCursor.buildRecordChain()`
**File:** `core/src/main/java/io/questdb/griffin/engine/window/CachedWindowRecordCursorFactory.java`

The window function execution has this flow:

```
Step 1: MATERIALIZE all source records into RecordArray
  while (baseCursor.hasNext()):
    offset = recordChain.put(record)
    for each ordered group:
      tree.put(chainRecord, ...)   // insert into red-black tree for ordering

Step 2: PASS 1 -- accumulate/compute
  For ordered functions:
    for each tree (order group):
      cursor = tree.getCursor()     // traverses in ORDER BY order
      while (cursor.hasNext()):
        offset = cursor.next()
        recordChain.recordAt(record, offset)
        for each function in group:
          function.pass1(record, offset, recordChain)

  For unordered functions:
    for each function:
      if (forwardScan):
        recordChain.toTop()
        while (recordChain.hasNext()):
          function.pass1(record, record.getRowId(), recordChain)
      else: // backward scan
        recordChain.toBottom()
        while (recordChain.hasPrev()):
          function.pass1(...)

Step 3: PREPARE PASS 2 (only for TWO_PASS functions)
  for each two-pass function:
    function.preparePass2()     // finalize aggregate, sort, compute result

Step 4: PASS 2 -- write results (only for TWO_PASS functions)
  Same iteration order as pass1, but calls function.pass2(record, offset, spi)
  pass2 writes the computed result into the RecordArray at the column position.
```

### Pass Count Semantics

| Value | Meaning | Use Case |
|-------|---------|----------|
| `ZERO_PASS` | Function computes result on-the-fly during pass1; pass1 writes directly to RecordArray | Running aggregates, sliding windows |
| `ONE_PASS` | Default. Compute during pass1, result read via getter during result iteration | Simple computations |
| `TWO_PASS` | pass1 accumulates state, preparePass2 finalizes, pass2 writes results | Whole-partition aggregates (stddev, percentile) |

### Reference: StdDevOverPartitionFunction (TWO_PASS)
**File:** `core/.../functions/window/AbstractStdDevDoubleWindowFunctionFactory.java` (line ~393)

```java
pass1(Record record, long recordOffset, WindowSPI spi):
  // Accumulates into per-partition Map using Welford's online algorithm
  // Map stores: [0]=mean, [1]=m2, [2]=count
  // Does NOT write to spi -- results not known yet

preparePass2():
  // Iterates all map entries
  // Computes final stddev from (m2, count) for each partition
  // Stores result back in map[0] (reuses the mean slot)

pass2(Record record, long recordOffset, WindowSPI spi):
  // Looks up partition key in map
  // Reads pre-computed result from map[0]
  // Writes to: Unsafe.putDouble(spi.getAddress(recordOffset, columnIndex), result)
```

### Reference: StdDevOverWholeResultSetFunction (TWO_PASS)
**File:** Same as above (line ~1498)

```java
// Simpler variant: no partition map, just instance fields
pass1:  count++; accumulate mean/m2 via Welford
preparePass2:  stddev = computeResultWelford(m2, count, isSample, isSqrt)
pass2:  Unsafe.putDouble(spi.getAddress(recordOffset, columnIndex), stddev)
```

## Pattern 3: Percentile Window Function (nw_percentile branch) -- Current State and Issues

### PercentileContOverPartitionFunction
**File:** `core/.../functions/window/PercentileContDoubleWindowFunctionFactory.java` (on nw_percentile branch)

This uses TWO_PASS with `MemoryARW` for per-partition value lists:

```
pass1:
  On each finite value:
    if new partition: allocate [size=1, value] in MemoryARW
    if existing: allocate NEW block [size+1, copy old values, new value]
    store offset in Map
  
  BUG: This is O(N^2) for N values per partition because every append
  copies all previous values into a new block. Fix: append-in-place
  with capacity tracking (like GroupByDoubleList pattern).

preparePass2:
  For each partition in map:
    quickSort the value list
    Compute interpolated percentile
    Store result at listPtr+8

pass2:
  Look up result from map, write to spi
```

### MultiPercentile Window Functions (nw_percentile branch)

The `MultiPercentileContDoubleWindowFunctionFactory` has the same O(N^2) copy-on-append pattern AND manages a `DirectArray result` field that is not freed in reset/reopen/toTop.

**BUG:** `DirectArray result` leaks memory across cursor reuses. Must be freed in `reset()` and `close()`. The `reopen()` method closes `listMemory` but does not reopen it (Reopenable contract violated).

## Pattern 4: Memory Management Lifecycle for Window Functions

### Three Lifecycle Methods

| Method | When Called | What It Does | Memory Handling |
|--------|-----------|--------------|-----------------|
| `toTop()` | Cursor re-scan (same query execution) | Reset computation state, keep allocations | `memory.truncate()`, `map.clear()`, reset counters |
| `reset()` | Cursor close (between query executions) | Free native memory, reset state | `Misc.free(memory)`, `Misc.free(map)` |
| `reopen()` | Cursor reopen after close | Re-initialize freed resources | `map.reopen()`, re-allocate memory |

**Correct pattern from StdDevOverPartitionRangeFrameFunction:**
```java
toTop():   super.toTop(); memory.truncate(); freeList.clear();
reset():   super.reset(); memory.close(); freeList.clear();
reopen():  super.reopen(); stddev = Double.NaN; // map.reopen() in super
close():   super.close(); memory.close(); freeList.clear();
```

**BasePartitionedWindowFunction lifecycle:**
```java
toTop():   super.toTop(); Misc.clear(map);  // clear map entries, keep map allocated
reset():   Misc.free(map);                   // free map's native memory
reopen():  map.reopen();                     // re-allocate map's native memory
close():   super.close(); Misc.free(map); Misc.freeObjList(partitionByRecord.getFunctions());
```

### DirectArray Lifecycle

`DirectArray` owns its memory via `Unsafe.malloc`/`Unsafe.realloc`/`Unsafe.free`:
```java
close():   ptr = Unsafe.free(ptr, size, MEM_TAG); // frees memory, sets ptr=0
clear():   // resets shape/strides/flatViewLength but does NOT free memory
ofNull():  // resets type to NULL, clears shape/strides, does NOT free memory
```

**CRITICAL:** `DirectArray.close()` frees the native memory. If a function field holds a DirectArray, it MUST call `Misc.free(directArray)` in both `reset()` and `close()`. The `ofNull()` method does NOT free memory -- it only resets metadata. If the caller replaces a live DirectArray's type/shape without ensuring the buffer is still valid, it creates a metadata/buffer mismatch.

## Pattern 5: Function Registration via function_list.txt

### How It Works

**File:** `core/src/main/resources/function_list.txt`

This file does NOT control which factories are loaded -- ALL factories implementing `FunctionFactory` are discovered via classpath scanning in `FunctionFactoryScanner`. The file controls **load ordering** (priority) when multiple factories match the same signature.

```
# Format: fully qualified class name, one per line
# Lines starting with # are comments
# Order in file = registration priority (earlier = higher priority)

# Example:
io.questdb.griffin.engine.functions.groupby.ApproxPercentileDoubleGroupByFunctionFactory
io.questdb.griffin.engine.functions.groupby.ApproxPercentileDoubleGroupByDefaultFunctionFactory
```

### What Needs Registration

Any new `FunctionFactory` class placed in `io.questdb.griffin.engine.functions.*` is automatically discovered. Adding it to `function_list.txt` is only needed when:
1. Multiple factories share the same function name with overlapping signatures
2. Load order matters for overload resolution

**For percentile functions:** The 10 missing group-by factories listed in PROJECT.md need entries in `function_list.txt` because they compete with existing factories (e.g., `percentile_disc(DD)` group-by vs `percentile_disc(DD)` window). Without entries, load order is undefined and the wrong factory may win.

### Signature Format

```
"function_name(TYPES)"
  D = double, L = long, I = int, S = string
  Uppercase = runtime parameter, Lowercase = constant parameter
  [] suffix = array type
  
Examples:
  "percentile_disc(DD)"    -- disc(double_col, double_percentile)
  "percentile_cont(DD)"    -- cont(double_col, double_percentile)
  "percentile_disc(DD[])"  -- disc(double_col, double_array_percentiles)
  "percentile_disc(LD)"    -- disc(long_col, double_percentile)
```

### isWindow() vs isGroupBy()

Factory methods that return `true` for `isWindow()` produce `WindowFunction` instances. Those returning `true` for `isGroupBy()` produce `GroupByFunction` instances. A factory cannot be both. The compiler uses this to route the function to the appropriate execution engine.

## Pattern 6: Off-Heap Memory for Window vs Group-By Functions

### Window Functions: MemoryARW

Window functions that need per-partition off-heap storage use `MemoryARW`:

```java
// Allocation
MemoryARW memory = Vm.getCARWInstance(
    configuration.getSqlWindowStorePageSize(),
    configuration.getSqlWindowStoreMaxPages(),
    MemoryTag.NATIVE_CIRCULAR_BUFFER
);

// Usage in pass1
long offset = memory.appendAddressFor(size) - memory.getPageAddress(0);
memory.putLong(offset, value);
memory.putDouble(offset + 8, doubleValue);

// Reading
double d = memory.getDouble(offset + 8);

// Lifecycle
toTop():  memory.truncate()   // reset append position, keep pages
reset():  memory.close()      // free all pages
reopen(): // must re-allocate via Vm.getCARWInstance or reopen()
close():  memory.close()
```

**Key insight:** `appendAddressFor()` returns an absolute address. Subtracting `getPageAddress(0)` gives a relative offset. The relative offset is stored in the Map. This is necessary because `getPageAddress(0)` can change if the memory is resized/relocated.

### Group-By Functions: GroupByAllocator

Group-by functions use `GroupByAllocator` (set via `setAllocator()`):

```java
// GroupByDoubleList wraps allocator
public void setAllocator(GroupByAllocator allocator) {
    listA.setAllocator(allocator);
    listB.setAllocator(allocator);
}

// The allocator is owned by the group-by execution engine, not the function.
// Functions just use it. cleanup is done by the engine.

// Function lifecycle:
clear():  listA.resetPtr(); listB.resetPtr();  // detach flyweights
close():  Misc.free(arg); Misc.free(percentileFunc);  // free child functions
// No allocator.free() calls -- engine manages allocator lifecycle
```

## Pattern 7: quickSort Duplication Problem

The nw_percentile branch has 8 copies of quickSort across 4 factory files:
- `PercentileContDoubleWindowFunctionFactory` (2 inner classes, each with quickSort+partition+swap)
- `PercentileDiscDoubleWindowFunctionFactory` (2 inner classes, each with quickSort+partition+swap)
- `MultiPercentileContDoubleWindowFunctionFactory` (2 inner classes)
- `MultiPercentileDiscDoubleWindowFunctionFactory` (2 inner classes)

All operate on `MemoryARW` with the same structure: median-of-three pivot, Lomuto partition (strict `<`).

**Fix approach:** Extract to a static utility class (e.g., `MemoryARWSort`) with signatures:
```java
static void quickSort(MemoryARW memory, long baseOffset, long left, long right)
static long quickSelect(MemoryARW memory, long baseOffset, long left, long right, long k)
```

The Lomuto `<` partition degrades to O(N^2) on all-equal elements. Use three-way partition (Dutch National Flag) or switch to quickSelect for single-percentile cases.

## Build Order Dependencies

Based on the architecture analysis, the build order for hardening is:

```
1. FOUNDATION (no dependencies)
   - Extract quickSort/quickSelect utility from duplicated copies
   - Fix setEmpty() in disc group-by functions (one-line fix each)
   - Register missing factories in function_list.txt

2. CORRECTNESS (depends on #1)
   - Fix O(N^2) copy-on-append in window pass1 (use capacity-tracking append)
   - Fix quickSort O(N^2) on all-equal elements (three-way partition)
   - Add percentile validation in multi-approx_percentile factories
   - Validate percentile argument is constant in window factories

3. MEMORY SAFETY (depends on #2)
   - Fix DirectArray leak in Multi* window reset/reopen/toTop
   - Fix zero-GC violation: pre-allocate double[] in Multi* preparePass2
   - Fix Reopenable contract in PercentileContOverPartitionFunction.reopen()

4. COMPLETENESS (depends on #3)
   - Add quantile_disc/quantile_cont window function aliases
   - Fix toPlan() to show percentile argument in EXPLAIN output
   - Add quickSelect to GroupByLongList for O(n) percentile on longs

5. TESTING (can partially overlap with #2-#4)
   - Switch existing tests from assertSql to assertQueryNoLeakCheck
   - Add window function tests for PARTITION BY paths
   - Add error-path tests (ORDER BY in window should fail)
   - Add tests for quantile_cont/quantile_disc SQL syntax
```

The critical ordering constraint: the shared utility (#1) must exist before fixing the quickSort bug (#2), because fixing it in 8 copies is error-prone. Memory safety (#3) must follow correctness (#2) because the memory patterns change when copy-on-append is fixed.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Copy-on-Append in MemoryARW
**What:** Allocating a new block and copying all old values on every append (current nw_percentile pattern).
**Why bad:** O(N^2) time and O(N^2) memory for N values per partition. A 100K-row partition does 5 billion copies.
**Instead:** Track capacity alongside size in the Map. Grow with doubling when full (same as GroupByDoubleList).

### Anti-Pattern 2: ofNull() as reset for DirectArray
**What:** Calling `ofNull()` on a DirectArray and expecting it to be reusable.
**Why bad:** `ofNull()` clears type and shape but does NOT free memory. The next `setType/setDimLen/applyShape` may see stale metadata from a previous call if type was already NULL.
**Instead:** Either `clear()` (keeps memory, resets metadata) or `close()` (frees everything).

### Anti-Pattern 3: Java Heap Array Allocation in preparePass2
**What:** `new double[size]` in `preparePass2()` of Multi* window functions.
**Why bad:** Violates zero-GC constraint. `preparePass2()` runs on the query data path.
**Instead:** Pre-allocate a reusable `double[]` field or use off-heap memory.

### Anti-Pattern 4: Missing reset()/reopen() for Owned Resources
**What:** A window function owns a `DirectArray` or `MemoryARW` but doesn't free it in `reset()`.
**Why bad:** Memory leaks across cursor reuses. The `assertMemoryLeak()` test wrapper catches these.
**Instead:** Every field that is `Closeable` must be handled in `close()` AND `reset()`. Fields handled in `reset()` must be re-created in `reopen()`.

## Scalability Considerations

| Concern | Small (1K rows) | Medium (100K rows) | Large (1M+ rows) |
|---------|-----------------|---------------------|-------------------|
| Copy-on-append | Negligible | 10B copies, seconds | 1T copies, minutes+ |
| quickSort all-equal | O(N log N) | O(N^2), seconds | O(N^2), minutes |
| DirectArray leak | 1 leak/query | 1 leak/query | 1 leak/query, but larger buffers |
| Off-heap per partition | ~8KB | ~800KB | ~8MB per partition |
