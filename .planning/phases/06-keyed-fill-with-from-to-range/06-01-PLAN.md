  <name>Task 2: Fix SIGSEGV crash for zero-key keyed FROM/TO</name>
  <files>core/src/main/java/io/questdb/griffin/engine/groupby/SampleByFillRecordCursorFactory.java</files>
  <action>
Fix the SIGSEGV in `SampleByFillCursor.initialize()` that occurs when keyed FILL with FROM/TO has FROM after all data.

**Root cause**: Pass 1 iterates the base cursor and discovers zero keys (GROUP BY returns empty because all data is before FROM). After pass 1, lines 492-494 unconditionally set `keysMapRecord` and `keysMapCursor` from the empty map. Pass 2 enters the FROM/TO branch (line 518-520) and sets up the bucket range. Then `hasNext()` reaches the gap handling at lines 374-379 which calls `emitNextFillRow()`. `emitNextFillRow()` calls `keysMapCursor.hasNext()` which returns false (empty map), then falls through to advance the bucket and recursively calls `hasNext()`. This loops until `nextBucketTimestamp >= maxTimestamp`, but on each iteration it may also try to read `keysMapRecord.getValue()` if cursor state is inconsistent, causing the SIGSEGV.

**Fix**: In `initialize()`, after pass 1 key discovery, add a guard for the zero-key case:

```java
// After pass 1 (after line 485: keyCount = keyIdx)
if (keysMap != null) {
    // ... existing key discovery code ...
    keyCount = keyIdx;
    if (keyCount == 0) {
        // No keys discovered — empty GROUP BY output.
        // With FROM/TO, emit zero rows (no keys to fill).
        // Without FROM/TO, also emit zero rows (no data).
        baseCursorExhausted = true;
        maxTimestamp = Long.MIN_VALUE;
        nextBucketTimestamp = Long.MAX_VALUE;
        return; // Skip pass 2 entirely
    }
    // ... existing keyPresent allocation, toTop, keysMapRecord setup ...
}
```

Place the `keyCount == 0` check immediately after `keyCount = keyIdx` (line 485) and before the `keyPresent` allocation (line 486). When zero keys are found, set `maxTimestamp = Long.MIN_VALUE` and `nextBucketTimestamp = Long.MAX_VALUE` so that `hasNext()` immediately returns false. Set `baseCursorExhausted = true`. Return early from `initialize()` to skip pass 2 entirely.

This produces an empty result set for keyed queries where no keys exist in the data range, which matches the cursor-path behavior (per CONTEXT.md decision: fill all keys from FROM, but zero keys means zero fill rows).

After the fix, run all tests including test 6 (testFillNullKeyedFromToAfterData) to verify the SIGSEGV is gone and the result is an empty set.
