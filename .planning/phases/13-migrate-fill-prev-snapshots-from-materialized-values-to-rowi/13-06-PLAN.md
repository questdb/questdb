**Defect 3 — Insufficient fill values grammar rule (RESEARCH.md §4.2)**

Current behavior on our branch: `generateFill()` silently pads missing fill expressions with `null` (line ~3445-3447: `fillValuesExprs.size() == 1 ? fillValuesExprs.getQuick(0) : null`). Query with 7 aggregates and `FILL(PREV, PREV, PREV, PREV, 0)` (5 specs) runs without error, producing null fill for the last 2 aggregates.

Master's behavior: throws `SqlException` with message "not enough fill values" (master points at position -1, we can do better — positioned at the fill clause).

Fix shape (RESEARCH.md §4.2, ~5 LOC):
After the per-column fill build loop completes (BEFORE the chain check), add:
```java
if (fillValuesExprs.size() > 1
        && fillValuesExprs.size() < aggNonKeyCount) {
    throw SqlException.$(fillValuesExprs.getQuick(0).position, "not enough fill values");
}
```
Where `aggNonKeyCount` is the number of non-key aggregate columns (exclusive of timestamp and key columns). Adjust the comparison to match the actual column-count variable used by the existing build loop — grep for the loop's iteration variable and the per-column check.

The single-element case (bare `FILL(PREV)` or `FILL(NULL)`) broadcast-fills all columns — this is the intentional behavior and must NOT trigger the error. The `fillValuesExprs.size() > 1` guard preserves that.
