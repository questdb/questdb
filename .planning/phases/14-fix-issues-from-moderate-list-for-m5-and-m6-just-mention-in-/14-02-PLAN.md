```java
@Test
public void testFillPrevKeyedBinary() throws Exception {
    // BINARY as key column. Prior to M-2, FillRecord.getBin/getBinLen had no
    // FILL_KEY branch, so gap rows returned null for the BINARY key. Post-fix:
    // keysMapRecord carries the key bytes through gap rows.
    //
    // Rendering strategy: BINARY test harness emits space-separated hex with a
    // 4-byte hex offset prefix (see PGJobContextTest.java:1289 for the canonical
    // precedent). Because rnd_bin() output is seeded-Rnd deterministic, we cannot
    // stably pin the expected hex output across JVM versions. Instead, this test
    // asserts the STRUCTURAL property we care about: gap rows' BINARY key is
    // NON-NULL (before the fix, it was null). The test materializes the query
    // result, iterates the records, and asserts the BINARY column on every row
    // (including fill rows) returns non-null via `record.getBin(colIdx) != null`
    // and `record.getBinLen(colIdx) > 0`.
    assertMemoryLeak(() -> {
        execute("""
                CREATE TABLE x AS (
                    SELECT
                        rnd_bin(4, 8, 0) AS k,
                        x AS v,
                        timestamp_sequence('2024-01-01T00:00:00.000000Z', 3_600_000_000L) AS ts
                    FROM long_sequence(1)
                ) TIMESTAMP(ts) PARTITION BY DAY""");
        // Second insert uses the SAME key bytes as the first row (captured via
        // cross-query reference). The simplest repeatable pattern: ingest two
        // rows with the same key by reading the first row's key back.
        execute("""
                INSERT INTO x
                SELECT k, 30.0, '2024-01-01T03:00:00.000000Z'::TIMESTAMP FROM x LIMIT 1""");
        //
        // The query `SAMPLE BY 1h FILL(PREV) ALIGN TO CALENDAR` produces 4 rows:
        //   00:00 (data, v=1.0), 01:00 (fill, v=prev 1.0), 02:00 (fill, v=prev 1.0),
        //   03:00 (data, v=30.0).
        // Pre-M-2 fix: the BINARY key `k` on the two fill rows is null.
        // Post-M-2 fix: the BINARY key `k` on the two fill rows is non-null AND
        // byte-equal to the data rows' key.
        //
        // Assert via a direct cursor-walk so the expected-output string does not
        // need to pin the hex rendering:
        try (RecordCursorFactory factory = select("""
                SELECT ts, k, first(v) FROM x SAMPLE BY 1h FILL(PREV) ALIGN TO CALENDAR""")) {
            try (RecordCursor cursor = factory.getCursor(sqlExecutionContext)) {
                Record record = cursor.getRecord();
                int rows = 0;
                while (cursor.hasNext()) {
                    rows++;
                    BinarySequence key = record.getBin(1);
                    long keyLen = record.getBinLen(1);
                    Assert.assertNotNull("row " + rows + " BINARY key must be non-null", key);
                    Assert.assertTrue("row " + rows + " BINARY key length must be > 0", keyLen > 0);
                }
                Assert.assertEquals("expected 4 rows (2 data + 2 fill)", 4, rows);
            }
        }
    });
}
```

This pattern avoids the rendering-stability problem while directly exercising the FILL_KEY branch of `getBin`/`getBinLen`.
