**Fix 1 — Infinite loop in hasNext() (SampleByFillRecordCursorFactory.java):**

Move the early-exit guard from line 317 to the top of the while loop body (immediately
inside `while (nextBucketTimestamp < maxTimestamp)`), BEFORE the code that computes
`dataTs`. The current guard at line 317 (`if (baseCursorExhausted && !hasExplicitTo)
return false`) is correct logic but unreachable: when `baseCursorExhausted` is true,
`dataTs` is `Long.MAX_VALUE`, which satisfies `dataTs > nextBucketTimestamp`, so the fill
branch fires and returns `true` before the guard is reached.

The fix is a single relocation. The new hasNext() structure should be:

```java
while (nextBucketTimestamp < maxTimestamp) {
    // Early exit: base cursor exhausted and no explicit TO bound —
    // stop emitting because there are no more data rows and no
    // trailing fill range was requested.
    if (baseCursorExhausted && !hasExplicitTo) {
        return false;
    }

    // Try to get the next data row's timestamp
    long dataTs;
    if (hasPendingRow) {
        // ... rest unchanged
```

Remove the old guard at line 317 (the `if (baseCursorExhausted && !hasExplicitTo)` block
and its comment "baseCursorExhausted and no TO clause — stop"). The logic is now at the
top of the loop, so the old position is dead code.

**Fix 2 — Sorted factory leak in generateFill() (SqlCodeGenerator.java):**

After line 3375 (`RecordCursorFactory sorted = generateOrderBy(...)`) add:
```java
groupByFactory = sorted;
```

This reassigns `groupByFactory` to point at the sorted wrapper so that the catch block
at line 3400 (`Misc.free(groupByFactory)`) frees the correct reference. When
`generateOrderBy` returns the same factory (no sort needed), this is a no-op. When it
wraps in a sort factory, `groupByFactory` now points to the wrapper, and `sorted` is the
same reference — both point at the wrapper that owns the original factory, so
`Misc.free(groupByFactory)` frees the entire chain.

Also update the `SampleByFillRecordCursorFactory` constructor call (line 3384-3398):
change `sorted` to `groupByFactory` in the metadata and base factory arguments (since
they are the same object after the reassignment).

**Fix 3 — constantFillFuncs leak in generateFill() (SqlCodeGenerator.java):**

In the catch block (lines 3400-3406), add `Misc.freeObjList(constantFillFuncs);` between
the existing `Misc.freeObjList(fillValues)` and `Misc.free(fillFromFunc)` lines.

Note: `constantFillFuncs` may contain `NullConstant.NULL` singletons. `Misc.free()` on
these is safe — `NullConstant` does not implement `Closeable`, so `Misc.free()` returns
immediately. Verify this by checking `NullConstant` — it extends `ConstantFunction` which
does not implement `Closeable`.

Declare `constantFillFuncs` before the try block (alongside `fillValues`) so it is
accessible in the catch block. Initialize it to `null` and create the actual `ObjList`
at line 3322 as before.
