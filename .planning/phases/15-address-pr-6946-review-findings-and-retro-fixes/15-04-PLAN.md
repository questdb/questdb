## 6c2c44237c - "Cover narrow-decimal FILL_KEY branches"

Date: 2026-04-20

Scope:
- +36 lines in `core/src/test/java/io/questdb/test/griffin/engine/groupby/SampleByFillTest.java` (single new test method `testFillKeyedDecimalNarrow`)
- Key column types covered: DECIMAL(2,0), DECIMAL(4,0), DECIMAL(9,0), DECIMAL(18,0)

## 2696df1749 - "Fix decimal128/256 sink null fall-through"

Date: 2026-04-21

Scope:
- Production change in `core/src/main/java/io/questdb/griffin/engine/groupby/SampleByFillRecordCursorFactory.java`:
  - `FillRecord.getDecimal128(int, Decimal128Sink)`: adds terminal `sink.ofRawNull()` after the FILL_CONSTANT branch (+16 lines / -3 lines against prior state, including the mirrored change for getDecimal256)
  - `FillRecord.getDecimal256(int, Decimal256Sink)`: same pattern as getDecimal128
  - `SampleByFillCursor.hasNext()`: `-ea` assert promoted on the defensive `dataTs < nextBucketTimestamp` branch, documenting an upstream-contract violation instead of silently early-returning
- Test changes in `core/src/test/java/io/questdb/test/griffin/engine/groupby/SampleByFillTest.java`: +64 lines across two new methods
  - `testFillKeyedPrimitiveTypes`: BOOLEAN / BYTE / CHAR / FLOAT / SHORT FILL_KEY coverage for primitives whose getters previously lacked explicit regression pins
  - `testFillPrevCrossColumnNoPrevYet`: cross-column PREV fall-through to default null when no prior data row exists

Rationale: The sink-caller contract requires a terminal `sink.ofRawNull()`
fall-through to mirror `NullMemoryCMR.ofRawNull`. Without it, callers that
reuse a Decimal128 or Decimal256 sink across rows would see stale prior-
row bytes instead of a clean null on FILL_CONSTANT miss. The `-ea` assert
makes upstream-contract-violation regressions surface loudly in test and
dev builds while the production fallback (silent skip) keeps shipping
query output uncorrupted. The two new test methods pin the new contract
and close a regression-coverage gap for primitive FILL_KEY paths.

Note: Phase 15 Plan 02 Task 2 extends the same fix to
`FillRecord.getLong256(int, CharSink<?>)`, which was overlooked in this
commit - the terminal `sink.ofRawNull()` and FILL_CONSTANT-branch explicit
`return` land there under M-4 closure.
